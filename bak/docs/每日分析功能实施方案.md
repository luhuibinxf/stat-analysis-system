# 每日分析功能实施方案

## 1. 系统架构

### 1.1 整体架构
- **前端**：HTML + Bootstrap + jQuery
- **后端**：C# Windows Forms 应用程序（内嵌 HTTP 服务器）
- **数据库**：SQL Server

### 1.2 核心组件
- **HTTP 服务器**：处理前端请求，提供 RESTful API
- **数据库操作层**：处理数据库连接和查询
- **业务逻辑层**：实现每日分析功能
- **前端界面**：提供用户交互界面

## 2. 数据库设计

### 2.1 新增表结构

#### 2.1.1 每日查询配置表 (`DAILY_QUERY_CONFIG`)
| 字段名 | 数据类型 | 约束 | 描述 |
|-------|---------|------|------|
| ID | INT | IDENTITY(1,1) PRIMARY KEY | 主键 |
| QUERY_NAME | NVARCHAR(100) | NOT NULL | 查询名称 |
| QUERY_TYPE | NVARCHAR(50) | NOT NULL | 查询类型 |
| QUERY_SQL | NVARCHAR(MAX) | NOT NULL | 查询SQL语句 |
| DESCRIPTION | NVARCHAR(255) | | 描述 |
| CREATED_BY | NVARCHAR(50) | | 创建人 |
| CREATED_TIME | DATETIME | DEFAULT GETDATE() | 创建时间 |
| UPDATED_BY | NVARCHAR(50) | | 更新人 |
| UPDATED_TIME | DATETIME | DEFAULT GETDATE() | 更新时间 |

#### 2.1.2 每日查询历史表 (`DAILY_QUERY_HISTORY`)
| 字段名 | 数据类型 | 约束 | 描述 |
|-------|---------|------|------|
| ID | INT | IDENTITY(1,1) PRIMARY KEY | 主键 |
| QUERY_DATE | DATE | NOT NULL | 查询日期 |
| QUERY_TYPE | NVARCHAR(50) | NOT NULL | 查询类型 |
| QUERY_PARAMS | NVARCHAR(MAX) | | 查询参数 |
| EXECUTION_TIME | DATETIME | DEFAULT GETDATE() | 执行时间 |
| EXECUTION_STATUS | NVARCHAR(20) | | 执行状态 |
| ROW_COUNT | INT | | 返回行数 |
| ERROR_MESSAGE | NVARCHAR(MAX) | | 错误信息 |

### 2.2 默认查询配置
系统预配置了以下查询类型：
- `system`：系统信息汇总
- `department`：执行科室汇总
- `category`：检查类别汇总
- `reporter`：报告医生工作量
- `reviewer`：审核医生工作量
- `technician`：技师工作量
- `status`：系统状态汇总
- `rank`：执行科室工作量排名

## 3. 功能实现

### 3.1 后端实现

#### 3.1.1 API 端点
- **POST /daily-analysis**：处理每日分析请求
  - 参数：`date`（查询日期）、`type`（查询类型）
  - 返回：JSON 格式的查询结果

#### 3.1.2 核心方法
- `GetDailyAnalysisData(string date, string type)`：获取每日分析数据
- `GetQuerySqlFromDatabase(SqlConnection conn, string type)`：从数据库获取查询SQL
- `GetDefaultQuerySql(string type)`：获取默认查询SQL
- `RecordQueryHistory(SqlConnection conn, string date, string type, int rowCount, string errorMessage)`：记录查询历史

### 3.2 前端实现

#### 3.2.1 界面组件
- 日期选择器：选择查询日期
- 分析类型下拉框：选择查询类型
- 查询按钮：触发查询
- 结果展示区域：展示查询结果

#### 3.2.2 核心函数
- `showDailyAnalysis()`：显示每日分析界面
- `runDailyAnalysis()`：执行每日分析查询
- `displayAnalysisResult(data, type)`：展示分析结果

## 4. 工作流程

### 4.1 初始化流程
1. 执行 SQL 脚本创建 `DAILY_QUERY_CONFIG` 和 `DAILY_QUERY_HISTORY` 表
2. 插入默认查询配置数据
3. 启动应用程序，初始化 HTTP 服务器

### 4.2 查询流程
1. 前端选择查询日期和分析类型
2. 前端发送 AJAX 请求到 `/daily-analysis` 端点
3. 后端接收请求，解析参数
4. 后端从数据库表 `DAILY_QUERY_CONFIG` 获取对应类型的查询 SQL
5. 如果数据库中没有对应查询，使用默认 SQL
6. 执行查询，获取结果
7. 记录查询历史到 `DAILY_QUERY_HISTORY` 表
8. 将结果转换为 JSON 格式返回给前端
9. 前端展示查询结果

## 6. 扩展性设计

### 6.1 新增查询类型
1. 在 `DAILY_QUERY_CONFIG` 表中添加新的查询配置
2. 前端下拉框会自动显示新的查询类型

### 6.2 自定义查询SQL
1. 编辑 `DAILY_QUERY_CONFIG` 表中的 `QUERY_SQL` 字段
2. 系统会自动使用新的查询SQL

### 6.3 性能优化
- 数据库索引：为常用查询字段添加索引
- 缓存机制：考虑添加查询结果缓存
- 异步处理：对于大数据量查询，考虑使用异步处理

## 7. 故障处理

### 7.1 数据库连接失败
- 系统会自动使用模拟数据，确保界面正常显示
- 记录错误日志到 `server.log` 文件

### 7.2 查询执行失败
- 系统会记录错误信息到 `DAILY_QUERY_HISTORY` 表
- 前端显示错误信息，确保用户体验

### 7.3 服务器异常
- 系统会自动重启 HTTP 服务器
- 记录错误日志到 `server.log` 文件
